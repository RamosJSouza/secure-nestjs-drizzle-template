# Auth Security Hardening Implementation Plan

> **For agentic workers:** REQUIRED: Use superpowers:subagent-driven-development (if subagents available) or superpowers:executing-plans to implement this plan. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Remedy the three confirmed vulnerabilities in `src/auth` and `src/users` (JWT token confusion, JTI revocation race, soft-delete without credential revocation) and close the related RLS/tenant-isolation gap.

**Architecture:** Add token-type discrimination (`typ`/`aud`/`iss`/`jti` on both tokens) so refresh tokens can't be used as access tokens; make JTI revocation atomic and fail-closed (gated on Redis availability) in the refresh rotation; make `UsersService.remove()` transactionally revoke all sessions and Redis JTIs (fail-closed in prod, fail-open in dev); add `organization_id` scoping to `UsersService.findAll`. RLS on `users`/`sessions` is **deferred** (Task 10) because it requires auth reads to be routed through `TenantDatabaseService.withTenant()` or a `BYPASSRLS` role first — enabling it now would break login/refresh/JwtStrategy. A shared helper `revokeSessionCredentials()` keeps revocation DRY across all six call sites.

**Tech Stack:** NestJS 11, Drizzle ORM 0.44 (node-postgres), `@nestjs/jwt` (RS256), `cache-manager` (Redis), Jest 30 (ts-jest), drizzle-kit. Scripts: `npm test`, `npm run db:generate`, `npm run db:migrate`, `npm run type-check`, `npm run lint`.

**Source vulns:** `security-audit/VULN-01-jwt-token-confusion.md`, `security-audit/VULN-02-jti-revocation-race.md`, `security-audit/VULN-03-softdelete-session-revocation.md`.

```ts
// src/auth/strategy/jwt.strategy.ts
import { ExtractJwt, Strategy } from 'passport-jwt';
import { PassportStrategy } from '@nestjs/passport';
import { Injectable, UnauthorizedException } from '@nestjs/common';
import { ConfigService } from '@nestjs/config';
import { UsersService } from 'src/users/users.service';
import { RequestContext } from '@/logger/request-context';
import { TokenRevocationService } from '@/security/token-revocation/token-revocation.service';
import { TOKEN_TYPE, TOKEN_ISSUER, TOKEN_AUDIENCE } from '../token-types';

@Injectable()
export class JwtStrategy extends PassportStrategy(Strategy) {
  constructor(
    private configService: ConfigService,
    private usersService: UsersService,
    private tokenRevocationService: TokenRevocationService,
  ) {
    super({
      jwtFromRequest: ExtractJwt.fromAuthHeaderAsBearerToken(),
      ignoreExpiration: false,
      secretOrKey: configService.get<string>('keys.publicKey'),
      algorithms: ['RS256'],
      issuer: TOKEN_ISSUER,
      audience: TOKEN_AUDIENCE,
    });
  }

  async validate(payload: { sub: string; jti: string; typ: string }) {
    if (payload.typ !== TOKEN_TYPE.ACCESS) {
      throw new UnauthorizedException('Invalid token type');
    }
    if (!payload.jti) {
      throw new UnauthorizedException('Token missing jti');
    }

    const revoked = await this.tokenRevocationService.isRevoked(payload.jti);
    if (revoked) {
      throw new UnauthorizedException('Token has been revoked');
    }

    const user = await this.usersService.findById(payload.sub);
    if (!user || !user.isActive) {
      throw new UnauthorizedException('Invalid token');
    }

    const now = new Date();
    if (user.lockedUntil && user.lockedUntil > now) {
      throw new UnauthorizedException('Account is locked. Try again later.');
    }

    RequestContext.setUser(user.id, user.organizationId ?? undefined);

    return user;
  }
}
```

## Task 6: DRY revocation helper `revokeSessionCredentials()`

**Why:** With refresh tokens now carrying `jti`, every place that revokes a session must also revoke **both** JTIs (access + refresh) in Redis. Otherwise a stolen refresh token used as an access token (pre-fix tokens aside) survives logout/password-change. Centralize it.

This is a non-behavioral signature widening that must land BEFORE the helper is added, otherwise `tsc` fails with `TS2554: Expected 2 arguments, but got 3`.

```ts
  // token-revocation.service.ts
  async revokeMany(jtis: string[], ttlSeconds: number, failClosed = false): Promise<void> {
    const results = await Promise.allSettled(
      jtis.map((jti) => this.revokeToken(jti, ttlSeconds)),
    );
    const failures = results.filter((r) => r.status === 'rejected');
    if (failures.length > 0) {
      this.logger.error(`${failures.length}/${jtis.length} JTI revocations failed`);
      if (failClosed) {
        throw new Error(`${failures.length} JTI revocations failed (failClosed)`);
      }
    }
  }
```

```ts
  private async revokeSessionCredentials(
```

```ts
// src/users/users.service.ts (add imports)
import { and, eq, isNull, sql } from 'drizzle-orm';
import { DatabaseService } from '@/database/database.service';
import { users, User } from '@/database/schema/users.schema';
import { sessions } from '@/database/schema/sessions.schema';
import { TokenRevocationService } from '@/security/token-revocation/token-revocation.service';
import { CreateUserDto } from './dto/create-user.dto';

@Injectable()
export class UsersService {
  constructor(
    private readonly dbService: DatabaseService,
    private readonly tokenRevocationService: TokenRevocationService,
  ) {}

  // ... existing methods unchanged ...

  async remove(id: string): Promise<void> {
    const failClosed = this.tokenRevocationService.isFailClosedEnabled();

    // Soft-delete + session revocation + Redis JTI revocation all inside ONE
    // transaction. If revokeMany fails in fail-closed mode, the rejection
    // propagates and drizzle rolls back the soft-delete (so the user is NOT
    // left half-deleted with live credentials). In fail-open mode (DISABLE_REDIS=true),
    // revocation failure is logged but the soft-delete commits — bounded by the
    // 15-minute access-token TTL.
    await this.dbService.db.transaction(async (tx) => {
      await tx
        .update(users)
        .set({ deletedAt: new Date(), isActive: false })
        .where(eq(users.id, id));

      const revoked = await tx
        .update(sessions)
        .set({ revokedAt: new Date() })
        .where(and(eq(sessions.userId, id), isNull(sessions.revokedAt)))
        .returning({
          id: sessions.id,
          accessTokenJti: sessions.accessTokenJti,
          refreshTokenJti: sessions.refreshTokenJti,
        });

      const jtis = revoked
        .flatMap((s) => [s.accessTokenJti, s.refreshTokenJti])
        .filter((j): j is string => !!j);

      if (jtis.length > 0) {
        try {
          await this.tokenRevocationService.revokeMany(
            jtis,
            TokenRevocationService.ACCESS_TOKEN_TTL_SECONDS,
            failClosed,
          );
        } catch (err) {
          if (!failClosed) {
            // fail-open: log and let the tx commit. Soft-delete still revokes
            // sessions in DB; only the Redis JTI list is stale (bounded by TTL).
            return;
          }
          throw err; // fail-closed: propagates -> tx rolls back
        }
      }
    });
  }
}
```

## Task 10 (DEFERRED): Enable RLS on `users` and `sessions` — requires prerequisite

> **DO NOT execute this task in this plan.** Enabling RLS on `users`/`sessions` **breaks the entire auth flow** unless a prerequisite is met first. This task is documented here so the engineer understands why it's deferred and what must happen first.

**Why deferred (the trap):** The session variable `app.current_tenant` is set **only** inside `TenantDatabaseService.withTenant()` (`src/tenant/tenant-database.service.ts:29`), which uses `SET LOCAL`. Neither `UsersService` nor the auth path (`login`, `refresh`, `JwtStrategy.validate`) use `withTenant()` — they query the global `db` without `set_config`. The `JwtStrategy` *sets* the tenant context **from** the user (chicken-and-egg: it needs the user row first).

If RLS is enabled on `users`/`sessions` now, every auth query runs with `current_setting('app.current_tenant', true) = NULL` → `organization_id::text = NULL` → `NULL` → **all rows filtered out**:
- `login` → `findOne` returns `undefined` → "Invalid credentials" for everyone.
- `refresh` → `UPDATE ... WHERE refresh_token_hash = H` affects 0 rows → "Invalid or expired refresh token".
- `JwtStrategy.validate` → `findById` returns `undefined` → 401 for every request.

**Prerequisite (must land in a separate plan BEFORE this task):**
1. Either route auth reads through `TenantDatabaseService.withTenant()` with a bootstrap path for login (where no user/context exists yet), **or**
2. Run the auth service DB connection as a Postgres role with `BYPASSRLS` (e.g., a dedicated `auth_role`), so auth queries ignore RLS while admin/list queries (via a tenant-scoped role) are isolated.

**SQL to apply ONLY after the prerequisite is in place** (`src/database/rls/0002_users_sessions_rls.sql`):

